**Repository**: https://github.com/Starryyu77/aicoach-codex

**Domain**: A Hermes-based fitness training Agent extension: fitness Skill Pack + UI Gateway + Training Cockpit + voice/video adapters + structured training cards.

## Actual requirement

The actual user need is not "build a fitness app." The goal is to validate a **training-process AI Agent extension**:

1. User can start a training conversation from a training cockpit UI.
2. User can speak or type short natural-language feedback during training.
3. Gateway sends the event, current plan, current exercise, recent training cards, memory summary, and time context to real Hermes.
4. Hermes returns structured JSON, not arbitrary natural language.
5. Frontend renders that structured output as plan cards, current exercise cards, training patches, history cards, and memory candidates.
6. User can backfill past workouts, plan future workouts, review previous sessions, and see/edit/delete history records.
7. Long-term preferences, risks, equipment, and observations are confirmation-first memory candidates, not automatic transcript dumping.
8. The system should support one in-project Settings flow where a user configures Hermes/MiniMax, ASR such as Doubao, and Vision providers by filling base URL/model/API key; the frontend must never hold plaintext API keys.

Hard boundaries:

- No database.
- No traditional backend.
- No SaaS admin.
- No coach workstation.
- No CRM, scheduling, payment, membership, or multi-student management.
- Do not rewrite Hermes Runtime, Memory, Sessions, or Skills.
- Do not use local Mock Hermes or local template replies for training dialogue. Production `/chat` must use real Hermes/LLM output or return a controlled error.

## Known design and implementation concerns

Please review these carefully; they are the reason for this handoff.

1. **Interaction model still feels rough.** The user expects natural conversation with a training agent, while the frontend also needs structured UI state. We currently rely on Hermes structured JSON plus Gateway UI mapping. Need advice on how to make this feel like a real training cockpit, not a fragile chat-to-card mapper.
2. **Plan generation logic is partly prompt/skill-driven and partly Gateway-assisted.** Gateway builds `exercise_selection_context`, but Hermes makes the final decision. We removed local template replies. Please review whether this split is appropriate or whether a clearer deterministic planning tool should exist behind Hermes.
3. **A2UI-inspired dynamic UI is early.** The current `AgentUiDocument` is allowlisted and safe, but it is not yet a full UI generation system. We want guidance on whether to deepen this approach or simplify.
4. **Memory layer is not true Hermes Memory yet.** The UI has pending updates and a display snapshot; long-term memory is meant to belong to Hermes. The project needs a cleaner bridge between Hermes Memory and local display/cache.
5. **File-backed state is intentionally lightweight but could become messy.** Current session, current plan, training cards, memory display, and provider config are local files. Need review of boundaries so this does not accidentally become a hidden database.
6. **Provider setup is integrated but operationally fragile.** Settings can save provider config and secrets by reference, but Hermes runtime setup still depends on local Hermes being available and correctly configured.
7. **Fitness methodology layer needs expert review.** We added NASM/ACE/NSCA/ACSM/RPE-RIR modules and official-source traces. Need review on how to make these useful and credible without turning the UI into citation spam.
8. **Frontend design quality is not yet good enough.** It is functional, not polished. Need recommendations for cockpit layout, plan/current-action synchronization, history editing, memory snapshot, and settings UX.

### Training methodology / tutorial references to integrate

The project aims to explicitly integrate these frameworks:

- NASM OPT Model:
  - https://www.nasm.org/certified-personal-trainer/the-opt-model
  - Used for phase/progression/regression logic.
- ACE Integrated Fitness Training Model:
  - https://www.acefitness.org/about-ace/ace-ift-model/
  - Used for client-centered assessment, preferences, confidence, and adherence.
- NSCA Essentials / Program Design:
  - https://www.nsca.com/certification/nsca-cpt/essentials-of-personal-training--3rd-edition/
  - Used for session structure, training frequency, recovery windows, exercise order, and total load.
- ACSM 2026 Resistance Training Guidelines update:
  - https://acsm.org/resistance-training-guidelines-update-2026/
  - Used for resistance-training variables by outcome: hypertrophy, strength, endurance, power, and function.
- RPE / RIR / Autoregulation:
  - Used for live load/reps/sets/rest adjustment based on user feedback, movement quality, and fatigue.
  - Current repo encodes this in `framework_autoregulation.md`; please recommend a stronger official or peer-reviewed source set if needed.
