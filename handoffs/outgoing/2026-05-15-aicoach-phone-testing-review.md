## Main Changes To Review

The branch adds a phone-first route and state fixes:

- `/phone` mobile cockpit with `PhoneTrainingCockpit`, `PhoneThread`, `PhonePlanDock`, `PhoneComposer`, quick actions, voice button, camera entry, and muscle picker.
- Chat transcript persistence: `current_session.chat_messages` is now saved by Gateway and hydrated by both `/phone` and `/training`.
- Plan patch persistence: Hermes `plan_patch` now updates `current_plan.json` rather than only rendering a reply card.
- Session reset/new conversation: clears current plan and chat state so refresh does not return to the previous conversation.
- Session end: saves normalized training cards to history and hides ended active plans from `/session/current`.
- History normalization: MiniMax-style fields like `completed_exercises` are normalized to `actual_completed`.
- Date normalization: saved plan/card/history fields use absolute dates instead of relative labels like today/tomorrow/yesterday.
- Source attribution: professional sources are now a deterministic backend enrichment path, including ACE IFT, NASM OPT, NSCA Program Design, ACSM 2026, and RPE/RIR.
- No local fallback generator is allowed. Hermes/provider failures must be surfaced or fixed on the real provider path.

## Constraints

- The reviewer will not have access to local files, `.runtime/`, local state, or API keys.
- Do not require live MiniMax/Hermes secrets for the core test plan. Live provider tests can be a separate optional tier.
- Do not propose adding a local fallback plan generator. The user explicitly rejected fallback behavior.
- Preserve the product boundary: phone is a frontend/control surface; execution remains local through Gateway and providers.
- Memory writes should remain confirmation-first. Do not silently persist arbitrary chat as long-term memory.
- The current storage model is intentionally lightweight file-backed state, not a database.

## Specific Concern

We need a rigorous testing and review strategy for the mobile/phone UI plus Gateway state contract. The current code has unit/integration tests, but we want an external reviewer to design a broader plan that can catch regressions like:

- navigating from `/phone` to `/history` and back loses chat messages;
- a plan modification appears in chat but does not modify the current plan card;
- refresh restores an ended plan;
- history cards fail to save because provider output shape changed;
- duplicate React keys from malformed cards or object-shaped quick actions;
- relative date labels are persisted instead of absolute dates;
- professional source traces are missing from generated plans;
- provider errors get hidden by unintended fallback behavior;
- mobile dock/composer overlays block content on small viewports.
