## Project background

**Project**: Road to Summer / 盛夏之路 - Fitness Training Agent  
**Repository**: https://github.com/Starryyu77/aicoach-codex  
**Current commit**: `0443375 Build local fitness agent prototype`  
**Domain**: A local-first AI Agent / Skill / Workflow for training-session planning, in-session adjustment, post-session logging, and confirmation-first memory.

This is not intended to be a traditional fitness app, coach workstation, SaaS backend, CRM, scheduling system, payment system, or membership system. The first milestone is to validate whether a training-process Agent can read local memory, understand short voice/text inputs during workouts, generate structured plans, adjust around equipment/fatigue/action feedback, and produce reusable training logs.

## What we have built

The repository now contains a dependency-free Node.js prototype that can run inside Codex or from the terminal. It implements the first deterministic version of the training Agent workflow:

```text
Voice/Text Input
  -> Transcript Cleaner
  -> Terminology Corrector
  -> Event Classifier
  -> Memory Reader
  -> Context Builder
  -> Workflow Router
      -> Preworkout Plan Generator
      -> Training Session Adjuster
      -> Post-session Log Writer
  -> Output Formatter
  -> Memory Writer Candidate
```

The prototype does not call an LLM yet. It deliberately starts with deterministic routing and output contracts so later model integration can be tested without losing the core workflow semantics.

## Implementation overview

### Input parser

Main file: `src/input/parseTrainingEvent.mjs`

The parser:

- Cleans ASR-like text.
- Normalizes common fitness terms.
- Extracts equipment, body parts, time, location, status, disliked items.
- Classifies input into events:
  - `training_request`
  - `daily_status`
  - `equipment_status`
  - `equipment_occupied`
  - `exercise_feedback`
  - `fatigue_feedback`
  - `pain_feedback`
  - `preference_update`
  - `location_memory_update`
  - `crowd_observation`
  - `plan_adjustment_request`
  - `training_completion`

Example:

```json
{
  "event_type": "equipment_status",
  "equipment": "高位下拉",
  "status": "unavailable",
  "scope": "today",
  "impact": "requires_substitution",
  "should_update_memory": true
}
```

### Memory system

Main files:

- `src/memory/loadMemory.mjs`
- `src/memory/memoryWriter.mjs`
- `memory/*.md`

The memory system is intentionally file-based. No complex database is used. Long-term memory writes are confirmation-first:

- Equipment status -> `equipment_memory.md`
- Location constraints -> `location_memory.md`
- Crowd observations -> `observation_memory.md`
- User preferences -> `preference_memory.md`
- Pain or risk -> `risk_memory.md`
- One-session details -> `training_logs.md`

Important rule: crowd and queue observations are not immediately promoted into hard rules. They first go to `observation_memory.md`.

### Context builder

Main file: `src/agent/contextBuilder.mjs`

It reads all memory files and builds the context object used by the current deterministic response generator. It currently has simple extraction logic, but it establishes the intended boundary: plan generation must be grounded in memory and not just in chat text.

### Response generation

Main files:

- `src/agent/respond.mjs`
- `src/agent/renderResponse.mjs`

The current implementation is deterministic. It produces:

- Structured training plan.
- Equipment replacement suggestions.
- Equipment memory update candidates.
- Fatigue decision framework that does not immediately stop training.
- Plain-language exercise cue for lat pulldown / back engagement.
- Crowd observation memory candidate.
- Training log card skeleton.

The next model adapter should attach here, not replace the parser/memory/router contracts.

### Codex skill entrypoint

Main file: `skills/training-process-assistant/SKILL.md`

This defines the local skill scope and boundary:

- Preworkout planning.
- In-session adjustment.
- Post-session log cards.
- Markdown / JSON memory reads.
- Confirmation-first memory candidates.

It explicitly excludes coach workstation, SaaS backend, CRM, membership management, scheduling, payment, and full app scope.

## Acceptance scenarios currently covered

The automated tests cover these seven meeting scenarios:

1. Preworkout plan generation from `今天该练什么？`.
2. Equipment occupancy: `高位下拉和绳索划船有人了。`
3. Equipment broken/repaired memory candidates:
   - `高位下拉坏了，今天不能用。`
   - `高位下拉修好了。`
4. Fatigue handling without over-conservatism:
   - `我有点累了，还要不要继续？`
5. Plain-language cue:
   - `我感觉不到背阔肌发力。`
6. Crowd observation:
   - `晚上7点健身房人很多，经常排队。`
7. Post-session summary:
   - `今天练完了，帮我总结一下。`

## What we want external expert feedback or implementation help on

Please keep the project boundary narrow: Agent / Skill / Workflow only.

Recommended next modifications:

1. Add a model adapter after `src/agent/respond.mjs`.
   - Keep deterministic `parseTrainingEvent`, memory loading, and schema validation.
   - Let the LLM enrich plan wording and action-specific explanations.
   - Validate model output against `schemas/*.schema.json`.

2. Improve the parser without making it overbroad.
   - Add more ASR error patterns.
   - Expand Chinese fitness term normalization.
   - Improve numeric parsing for sets/reps/weights/RPE.
   - Keep tests for every new intent pattern.

3. Make `runtime/current_session.json` actually update.
   - Current code returns session updates but does not persist all in-session events.
   - Add append/update helpers for current session state.
   - Preserve event history in `runtime/parsed_events.jsonl`.

4. Implement confirmation flow for memory writes.
   - `memoryWriter.mjs` already creates candidates and has an allowed target list.
   - Add a CLI or API command such as `confirm-memory <candidate-id>`.
   - Do not allow arbitrary file paths from user text.

5. Add ASR integration behind `src/asr/asrStub.mjs`.
   - The interface is already isolated.
   - First real provider can be swapped without changing parser logic.

6. Add a lightweight local UI only after the workflow is stable.
   - Avoid turning this into a full app.
   - The UI should show parsed event, structured plan/adjustment, and memory candidates.
